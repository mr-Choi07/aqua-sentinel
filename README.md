# 어장지킴이 (Aqua Sentinel)
### 고수온·적조 조기경보 + 피해 증빙 자동화 AI

> 2026 부산·울산·경남 AI 활용 아이디어 공모전(나는 Solo AI) 출품작

## 1. 프로젝트 소개

통영·거제·남해·고성·하동의 해상 가두리·패류 양식 어가는 매년 여름 고수온·적조·빈산소로
막대한 피해를 입고 있습니다. 그런데 국립수산과학원의 특보는 '해역 단위'로만 발령되어
개별 어장·어종에 맞는 구체적인 행동 지시로 이어지지 못하고, 피해 신고와 증빙도 여전히
수작업에 의존하고 있어 정당한 보상이 새고 있습니다.

**어장지킴이**는 이 두 가지 격차 — ① 해역 특보와 개별 어장 사이의 **정보 번역 격차**,
② 피해 발생과 정당한 보상 사이의 **증빙 격차** — 를 메우기 위한 서비스입니다. 공공
API로 실시간 수온·적조 데이터를 수집해 어장·어종별 위험도를 산출하고, LLM 기반 대응
코치와 폐사 사진 기반 신고서 자동 생성 기능을 제공합니다. 고령 어업인이 스마트폰에서
바로 쓸 수 있도록 설치 없이 홈 화면에 추가할 수 있는 PWA(Progressive Web App)로
만들었습니다.

## 2. 왜 이 지역 문제인가

### 피해 규모

경남 지역 고수온 피해는 2021년 213개 어가·116억 5,900만 원에서 2023년 322개 어가·
207억 2,200만 원, 2024년 925개 어가·659억 2,200만 원(어류 2,396만 9천 마리 포함)으로
3년 새 어가 규모 약 4배, 피해액 약 5배로 늘었습니다.

2024년 여름에는 전국 멍게 생산량의 70%를 차지하는 남해안 양식 멍게가 통영·거제 지역을
중심으로 95% 이상 폐사했고, 이듬해까지 종묘 생산용 어미 멍게마저 전멸해 향후 2~3년간
공급 부족이 예고될 정도로 피해가 이어졌습니다. 통영·거제는 이 프로젝트가 대상으로 삼는
바로 그 지역입니다.

> 출처: 경남신문(2025.3.4) 종합 보도, 해양수산부 보도자료(mof.go.kr) 재난지원금 지급
> 현황 / MBC뉴스(2024.8.26), 국제신문(2024.8.26), 경남도민일보(2024.8월) 통영·거제
> 멍게 폐사 보도. 어가 수·피해액은 언론 보도 기준이며, 최종 확정 수치는 해양수산부/
> 경상남도 공식 통계를 별도 확인 권장.

### 구조적으로 왜 반복되는가

이 정도 피해 규모에도 문제가 반복되는 이유는 예보가 없어서가 아니라, 예보와 현장
사이에 두 단계의 격차가 있기 때문이라고 판단했습니다.

- **정보 번역 격차** — 고수온 특보는 '남해 중부 연안'처럼 넓은 해역 단위로 발령됩니다.
  같은 해역 안에서도 관측소별 수온 차이, 양식 어종별 폐사 임계수온 차이가 있어, 어업인
  입장에서는 "우리 어장이 지금 위험한 건지, 뭘 해야 하는지"를 특보 문구만으로 바로
  판단하기 어렵습니다.
- **증빙 격차** — 피해가 실제로 발생해도 신고서 작성, 사진 정리, 서식에 맞춘 문서화는
  전부 수작업입니다. 성수기에 여러 어장을 동시에 관리하는 어업인일수록 이 절차에 쓸
  시간이 부족하고, 이는 신고 누락·지연으로 이어져 정당한 재난지원금을 놓치는 결과를
  낳습니다.

어장지킴이는 이 두 격차 각각에 대응하는 기능(어장·어종별 위험도 번역, 신고서 자동
초안 생성)으로 구성되어 있습니다.

## 3. 주요 기능

1. **위경도로 내 어장 직접 등록** — 관측소 목록에서 고르는 대신, 위도·경도(또는 "내
   위치 사용" 버튼)와 양식 어종·입식량만 입력하면 최근접 관측소를 자동으로 매핑합니다.
   관측소 커버리지 밖(예: 하동 등)이면 거리 기준으로 "가장 가까운 관측소도 이만큼
   떨어져 있다"고 정직하게 알려줍니다 — 없는 걸 있는 것처럼 보여주지 않습니다.
2. **어장 맞춤 위험 번역** — 공공 API의 실시간 수온·적조 데이터를 관측소·어종별 임계
   수온과 비교해, 해역 단위 특보를 "내 어장이 지금 정상/관심/주의/경보 중 어디인지"로
   바꿔서 보여줍니다.
3. **LLM 기반 대응 코치** — 위험 등급과 관측 근거를 바탕으로, 공식 대응 매뉴얼에 근거한
   행동을 우선순위·긴급도가 매겨진 구조화된 형태로 안내합니다(자세한 내용은 4장 참고).
4. **피해 증빙 자동화** — 폐사 사진을 멀티모달 AI로 분석하고, 참고용 요약본과 실제
   정부 서식(자연재난 피해신고서)에 맞춘 제출용 PDF를 자동 생성합니다.
5. **능동 알림(웹 푸시)** — 로그인 없이 브라우저 알림 구독만으로, 위험 등급이 바뀔 때
   폰으로 직접 알려줍니다 — 앱을 직접 열어봐야만 상태를 알 수 있는 구조를 피했습니다.

## 4. AI를 어디에, 어떻게 썼는가

AI를 "썼다"는 사실보다, 정확히 어디에 쓰고 어디에는 안 썼는지가 더 중요하다고
판단해 명확히 구분했습니다.

### AI가 아닌 부분 — 위험도 산출

어장별 정상/관심/주의/경보 판정은 LLM 호출이 아니라 규칙 기반 로직입니다. 실시간
관측 수온을 어종별 임계값과 비교하고, 최근 관측 이력에 단순 선형회귀를 적용해 단기
추세를 근사합니다. 이 값은 국립수산과학원의 정밀 해양예보모델을 대체하는 것이 아니라,
그 특보가 나오기 전/후 공백을 메우는 보조 지표라는 점을 화면에도 명시합니다. 판단
과정이 100% 결정론적이어야 하는 부분에 LLM을 쓰지 않은 것은 의도적인 선택입니다.

### AI를 쓴 부분 1 — 대응 코치 (Claude Haiku 4.5)

관측 데이터(현재 수온, 등급, 판정 근거, 어종별 임계값, 최근 추세, 적조속보 해당 여부)를
구조화해 전달하고, 자유 서술이 아니라 고정된 JSON 스키마(상황 요약 1문장 + 우선순위가
매겨진 행동 최대 4개 + 필요시 문의처)로만 응답하도록 강제합니다. 실제로 개발 중 테스트
과정에서 아래 두 가지 문제를 발견해 방어 로직을 추가했습니다 — AI 응답을 그대로
신뢰하지 않고 검증한 사례로 남겨둡니다.

- 모델이 "국립수산과학원 홈페이지에서 수온을 확인하라"고 안내한 경우가 있었습니다.
  이 앱이 이미 실시간 수온을 보여주고 있어 앱의 존재 이유와 모순되는 답변이라, 시스템
  프롬프트에 이를 명시적으로 금지하는 규칙을 추가했습니다.
- 모델이 실존 여부를 확인할 수 없는 지자체 전화번호를 지어낸 경우가 있었습니다.
  향후 수온 전망 같은 예측성 수치 생성 금지 규칙과 함께, 프론트엔드에도 전화번호
  패턴을 걸러내는 코드를 추가해 이중으로 방어합니다.

### AI를 쓴 부분 2 — 폐사 사진 분석 (Claude Haiku 4.5, 멀티모달)

업로드된 폐사 사진에서 관찰되는 소견(추정 폐사 규모, 추정 원인, 전체 소견)을 구조화된
출력으로 받습니다. AI는 폐사 원인을 확정 진단하지 않고 "관찰되는 내용을 서술"하는
수준으로만 쓰고, 그 결과를 실제 정부 서식(자연재난 피해신고서)에 옮길 때도 담당
공무원이 현지 확인 후 기재해야 하는 항목(피해물량 확정, 피해 구분 등)은 시스템이
자동으로 채우지 않고 항상 공란으로 남깁니다. AI 초안이 실제 신고서로 오인되지 않도록,
생성된 PDF에는 눈에 띄는 워터마크를 넣습니다.

## 5. 기술 스택

- **Python 3.11+ / FastAPI** — 백엔드 API (pipeline/ 로직을 REST로 노출)
- **React + TypeScript + Vite + Tailwind CSS** — 프론트엔드, PWA(manifest + service worker)
- **SQLite** — 초기 DB (추후 필요 시 Postgres 전환 가능하도록 설계)
- **pandas / numpy** — 시계열 데이터 처리, 추세 선형회귀
- **requests** — 국립수산과학원(NIFS) Open API 연동
- **Anthropic API (Claude Haiku 4.5)** — 구조화 출력 기반 대응 코치, 멀티모달 폐사
  사진 분석
- **fpdf2 / pypdf / reportlab** — 참고용 신고서 PDF 생성, 실제 정부 서식 오버레이
- **pywebpush** — 웹 푸시 알림(VAPID)

## 6. 폴더 구조

```
aqua-sentinel/
├── backend/          # FastAPI 서버 (pipeline/ 을 REST API로 노출)
├── frontend/          # React + Vite PWA
├── pipeline/          # 공공 API 데이터 수집 + 위험도/코치/신고서/알림 로직
├── data/              # 수집된 데이터 저장 (git 추적 제외)
├── docs/              # 출품작 설명서 등 문서
├── requirements.txt
├── .env.example       # API 키 자리 표시용 (실제 키는 .env에 별도 보관)
└── .gitignore
```

## 7. 설치 및 실행 방법

```bash
# 1. 저장소 클론
git clone https://github.com/<your-id>/aqua-sentinel.git
cd aqua-sentinel

# 2. 백엔드 의존성 설치 (가상환경 권장)
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt

# 3. 환경변수 설정
cp .env.example .env
# .env 파일을 열어 NIFS_TEMP_API_KEY, NIFS_REDTIDE_API_KEY, ANTHROPIC_API_KEY,
# VAPID_PUBLIC_KEY/VAPID_PRIVATE_KEY(python -m pipeline.push로 생성) 값을 채워 넣기

# 4. 백엔드 실행 (포트 8010)
python -m uvicorn backend.main:app --port 8010 --reload

# 5. 프론트엔드 실행 (다른 터미널에서)
cd frontend
npm install
npm run dev
# http://localhost:5173 에서 확인
```

## 8. 현재 범위와 한계

3주라는 개발 기간 안에서, 넓게 벌리기보다 "조기경보 → 대응 코치 → 피해 증빙"으로
이어지는 한 흐름을 실제로 끝까지 동작하게 만드는 데 집중했습니다. 그래서 아래
제약이 있고, 공모전 심사에서도 숨기지 않고 그대로 안내할 계획입니다.

- **관측소 범위**: 통영·거제·남해·고성 11개 관측소를 다룹니다. 하동은 국립수산과학원
  실시간어장정보 API 응답에 해당 관측소가 없어(패류양식 위주 지역 특성으로 추정)
  제외되어 있습니다 — 어장 등록 시 하동 좌표를 넣으면, 없는 척하지 않고 "가장 가까운
  관측소도 이만큼 떨어져 있다"는 경고와 함께 최근접 관측소로 안내합니다.
  대체 데이터 소스도 조사했습니다: 국립수산과학원 자체 관측소 메타데이터에는 하동에서
  11.5km 떨어진 "남해 설천"이 있지만 실시간 피드에는 비활성 상태라 쓸 수 없었고,
  국립해양조사원(KHOA)이 별도로 운영하는 어장환경정보 서비스에 광양만(하동과 접한
  만) 관측점이 있어 이쪽이 더 유력한데, 신규 기관 API 키 신청(전자민원, 승인 대기
  필요)이 걸려 있어 **신청 진행 중**입니다. 승인되면 통합하고, 반려되거나 응답이
  지연되면 인접 관측소로의 매핑을 유지합니다.
- **관측소 좌표의 정확도 편차**: 어장 등록의 최근접 관측소 매핑에 국립수산과학원의
  관측소 메타데이터(위도·경도)를 그대로 썼는데, 그 중 "고성 가진" 1곳의 좌표가
  실제로는 강원도 고성(동해안) 좌표로 잘못 등록되어 있는 것을 개발 중 발견했습니다.
  이 관측소만 관측소 자체 좌표가 아니라 경남 고성군 행정구역 중심 좌표(위키데이터
  기준)로 대체했고, 코드에 그 이유를 남겨뒀습니다 — 나머지 10곳보다 정밀도가
  낮습니다.
- **어종별 임계수온 근거**: 우럭·멍게·전복은 개별 문헌 근거를 확인했습니다. 굴·참돔은
  개체별 임계수온을 규정한 학술 자료를 추가로 찾아봤지만 발견하지 못해 일반 기준을
  그대로 씁니다 — 다만 그 이유가 다릅니다. 굴은 오히려 "30℃ 이상에서도 생존 개체가
  보고"되어 있고 업계에서 폐사 원인을 수온보다 동반 빈산소수괴로 보는 시각이 우세해,
  단일 수온 임계치 모델 자체가 이 어종에는 안 맞을 수 있다는 걸 알면서 잠정 사용
  중입니다. 참돔은 실제 피해 보도가 일반 기준(28℃/3일)과 같은 온도대에서 반복
  관측돼 일반 기준을 쓰는 근거가 어느 정도 있습니다. 두 경우 다 수치를 바꿀 근거는
  아니었지만, 코드에 조사 결과와 출처를 남겨 "확인 못함"과 구분해뒀습니다.
- **빈산소 특보 미연동**: 표층-저층 온도차로 성층화 가능성만 근사하고 있고, 국립수산
  과학원의 공식 빈산소수괴 속보는 아직 연동하지 않았습니다.

다음 단계로는 하동 등 관측 공백 지역의 대체 데이터 소스 확보(국립해양조사원 API
